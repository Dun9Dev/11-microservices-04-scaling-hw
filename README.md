# Домашнее задание к занятию «Микросервисы: масштабирование»

Вы работаете в крупной компании, которая строит систему на основе микросервисной архитектуры.
Вам как DevOps-специалисту необходимо выдвинуть предложение по организации инфраструктуры для разработки и эксплуатации.

## Задача 1: Кластеризация

Предложите решение для обеспечения развёртывания, запуска и управления приложениями.
Решение может состоять из одного или нескольких программных продуктов и должно описывать способы и принципы их взаимодействия.

Решение должно соответствовать следующим требованиям:
- поддержка контейнеров;
- обеспечивать обнаружение сервисов и маршрутизацию запросов;
- обеспечивать возможность горизонтального масштабирования;
- обеспечивать возможность автоматического масштабирования;
- обеспечивать явное разделение ресурсов, доступных извне и внутри системы;
- обеспечивать возможность конфигурировать приложения с помощью переменных среды, в том числе с возможностью безопасного хранения чувствительных данных таких как пароли, ключи доступа, ключи шифрования и т. п.

Обоснуйте свой выбор.

## Решение 1:
### Решение: Kubernetes

### Сравнительная таблица решений для кластеризации

| Требование | Kubernetes | Docker Swarm | Nomad | Amazon ECS |
|------------|------------|--------------|-------|------------|
| Поддержка контейнеров | Да | Да | Да | Да |
| Обнаружение сервисов | Встроенное (DNS, Services) | Встроенное (DNS) | Через Consul | Встроенное |
| Маршрутизация запросов | Ingress, LoadBalancer | Встроенная | Через Fabio/Consul | Load Balancer |
| Горизонтальное масштабирование | Да | Да | Да | Да |
| Автоматическое масштабирование | Да (HPA) | Нет | Да | Да |
| Разделение внешних/внутренних ресурсов | Да (типы Service) | Да (сети) | Да | Да (Security Groups) |
| Конфигурация через переменные среды | ConfigMap, Secrets | Configs, Secrets | Variables | Environment variables |
| Безопасное хранение секретов | Secrets, Vault интеграция | Docker Secrets | Vault интеграция | Secrets Manager |

### Обоснование выбора Kubernetes:

1. **Промышленный стандарт** — де-факто стандарт оркестрации контейнеров, огромное сообщество

2. **Полное соответствие требованиям**:
   - Service Discovery через DNS и Services
   - Маршрутизация через Ingress и LoadBalancer
   - Автомасштабирование через HPA (по CPU, памяти, кастомным метрикам)
   - Разделение доступа через типы Service (ClusterIP — внутри, LoadBalancer/Ingress — снаружи)
   - Конфигурация через ConfigMap и Secrets

3. **Гибкость** — работает везде (on-prem, любые облака), нет привязки к вендору

**Альтернативы:**
- **Docker Swarm** — проще, но меньше возможностей
- **Nomad** — легковесный, но требует Consul для service discovery
- **ECS** — только для AWS, хорош если уже всё в AWS
